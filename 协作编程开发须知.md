# 改 council 协作编程之前,先读这些

> 这些是反复踩过的坑抽象出来的通用原则。改"协作编程 / 持续协作 / 终端"相关代码前先读一遍,避免重复犯。

## 一、用程序去操作交互式 CLI(claude / codex / gemini / grok…)

1. **文字和回车要分两次发,中间留一点延迟。** 一次性把"文字 + 回车"一起塞进去,回车会被终端的粘贴机制吞掉,内容会卡在输入框里不提交。
2. **改完一定实际跑一遍真 CLI,确认它真的提交、真的动了。** 不要只看代码逻辑对不对就收工——终端输入这类事,凭空假设十有八九是错的。
3. **绝不在对方正忙、或用户正在打字时插入输入。** 先确认它空闲(一段时间没新输出)、且用户最近没敲键盘,再发。
4. **周期性催它时,发最短的一句话就够。** 它启动时已经知道自己的职责,别每次重打一大段长指令。

## 二、让 AI 持续自动干活

5. **交互式 CLI 每干完一回合就会停下、交还控制——这是天性。** 想让它持续干,必须有东西定时再戳它;只给一次指令,它做完就停。
6. **催它的指令要把话说死:** "没有反馈就直接做下一步,不要停下来问我、不要等确认,直到整个任务真正完成。" 否则它会停在那等人。
7. **负责"实现 / 主写"的那个,应该一空闲就接着下一步**,不要让它按固定间隔干等。

## 三、多个 AI 配合 / 重启 / 进度

8. **多个 Agent 有先后依赖时,不要让它们同一时刻一起检测。** 下游会扫到上游还没改完的状态、差一拍。要各自错开时间(下游比上游晚)。
9. **进度要落到一个共享文件里**(谁做完哪步就记下来)。重启后让 Agent 先读这个文件接着上次干,**而不是从头重做**。

## 四、UI / 工作流

10. **一个功能别拆成两个控件,让其中一个看着像开关、其实没用。** "设了间隔"就该等于"开启",或者干脆默认开;别让用户以为开了、其实没开。
11. **不要静默覆盖之前的状态。** 留一份历史,让旧的东西能找回来。
12. **别每改一点就重新打包 / 重装**(会杀掉用户正在跑的程序、打断他)。攒一批再一起更新。

## 最重要的一条

> 上面这些坑,大多来自"想当然、没在真实环境里验证"。涉及终端输入、CLI 行为、定时、多 Agent 时序这类,**改完务必在真实环境实跑一遍看真实结果**,别只看代码。
