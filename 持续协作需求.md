# 多 AI 协作编程 · 持续协作 需求总结

> 把这一轮提的所有问题归整成"这个功能该怎么设计"。给设计/实现的人看，只讲该有的行为，不讲具体代码。

## 一、主写（实现者）的工作循环 —— 核心
每完成一步，按这个固定逻辑走，不许偏：

1. 先看 TEAM_NOTES.md
2. **有反馈** → 优先处理反馈
3. **没反馈** → 继续推进还没完成的任务（scope）
4. **scope 已全部完成** → 停止开发：
   - 不制造任务
   - 不重复编辑共享文件
   - 不为了"有动作"而做动作
   - 进入 **Idle**，等待新的需求 / 缺陷 / 反馈 / 方向

要点：**绝不停下来问人、不等人工确认**（没反馈就直接做下一步）；但也**绝不没活硬找活、造功能、超出 scope**。这两端都要防住。

## 二、节奏 / 等待
- 主写**写完一步后，先等 N 分钟**（N 用户可自由填，默认 2）留出人工介入窗口；这段时间**人一旦插手（打字）就别自动催**，听人的；没人介入才提示它继续。
- 审查、测试**各自设自己的巡检间隔**，不要和主写、也不要互相绑死。

## 三、多 AI 时序（防"差一拍"）
- 第 2、第 3 个职责不同、还可能有先后依赖。**不能让它们在同一时刻一起检测**——否则下游扫到的是上游"还没改完"的状态，白扫一轮、差一拍。
- 要能把**下游设得比上游晚**，错开时间点。

## 四、催 Agent 的机制（技术坑）
- 给交互式 CLI 发指令：**文字和回车要分开发**，否则会卡在输入框里不提交。
- **只在 Agent 空闲、且人没在打字时**才发，别插进它正忙 / 人正输入的时候。
- 催的话是**一句短的**，别每次重打一大段（它已经知道自己职责）。
- 催的措辞是"**检查，没事就保持 Idle**"，不是"必须干活"——既能收到队友后来写的新反馈，又不会被 ping 着硬造活。

## 五、控制（随时能管）
- **一键全部停止 / 全部开启**。
- **单独停 / 单独开某一个 AI**，按钮要**明显**（不是藏在角落的小图标）。
- "停" = 停掉它的自动循环 + 中断它当前正在做的动作；"开" = 恢复。

## 六、持久化 / 重启 / 历史
- **重启后能续上次，不从头重做**——Agent 开工前先看现有代码 + 进度记录，已完成的不重复；每完成一步把进度记下来，供重启后接着干。
- **任务配置要留历史**——换个新任务别把旧的覆盖了、找不回来；能从列表点回任意一份旧任务。

## 七、协作规则
- **出现争议不要盲目改、也不要互相来回覆盖**——先在共享文件（TEAM_NOTES.md）里写清理由和方案、协商、记录讨论，**达成共识、选大家最认同的最优方案**后再动手。

## 八、开关 / 默认（防踩坑）
- "持续协作"别拆成"一个开关 + 一个分钟数"，让人以为设了分钟就开了、其实没开。**设了间隔就该等于开启**，或者默认就开。
