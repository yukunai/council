# council 开发笔记 / 变更记录

> 项目的功能演进与关键决策记录（按时间倒序，最新在上）。

## 2026-06-01

### 协作编程：主写 scope 完成后进入 Idle，不再瞎造活
- 旧规则"没反馈就直接做下一步"会导致：没反馈→找下一步→没下一步→硬找→造功能→偏离 scope。
- 升级成五步：① 看 TEAM_NOTES.md；② 有反馈→优先处理；③ 没反馈→推进未完成任务；④ scope 已全部完成→**不制造任务、不重复编辑共享文件、不为"有动作"而动作，进入 Idle 等待新的需求/缺陷/反馈/方向**。
- 初始指令 + 循环 nudge 都改成这套。关键：**循环照常扫**(否则收不到队友后写的新反馈)，但 nudge 措辞是"检查 + 没事就 Idle"，不是"必须干活"。要彻底停某个 Agent 用标签上的 ■。

### 协作编程：有争议先协商、别盲改
- TEAM_NOTES.md 协作约定(`notes`，对所有 Agent 生效)加一条：对改动/反馈有不同意见、出现争议时，**不盲目直接改、不来回互相覆盖**；先在 TEAM_NOTES.md 写清理由和方案、和队友协商、把讨论与结论都记进去，达成共识选出最优(大家最认同)方案后再动手。

### 协作编程：停止/开启 AI 工作（单个 + 全部）
- 把 持续协作 循环逻辑从 runCoding 抽成 `startAgentLoop(term)`，config 存在 `Term.loopParams`，所以停了能再开。
- **单个 AI**：每个 Agent 标签上加 ■/▶ 切换——■ 停（清循环 + 发 Esc 中断当前动作）、▶ 开（按存的 config 恢复循环）。
- **全部**：终端区顶部 `#term-toolbar`（有 Agent 时才显示）两个按钮「■ 全部停止」「▶ 全部开启」。
- 循环计时器从 `number[]` 改成 `codeLoops: {timer, term}[]`，可按 term 精确停/查；CLI 退出时自动从数组摘除并刷新标签/工具栏。重开用 `lastFire=now-minGap`，恢复后下一个空闲窗口即触发。

### 协作编程：主写写完一步先休息 2 分钟、留人工介入窗口
- 之前主写空闲即续(15s)太急,没给人插手的机会。改:主写写完一步后**静默满 2 分钟(没新输出 + 没敲键盘)才提示继续**——这 2 分钟里人工一旦干预(打字)就不自动催、让用户来。审查/测试不受影响(各自间隔)。
- 实现：loop 里 isImpl 时 `minGapMs/QUIET/USER_QUIET` 全取 `REST_IMPL`。
- 这个等待时间**可自由填**：实现位卡片上「写完一步等 [N] 分钟没人介入再继续」(复用 `a.loopMins`，默认 2)，不再写死 2 分钟。

### 协作编程：每个 Agent 单独的巡检间隔 + 持续协作默认开
- 痛点①「持续协作」反复忘勾(loop=false),设了分钟数也不跑 → 修:**改全局分钟数自动开启持续协作** + `DEFAULT_CODE.loop` 默认 true。
- 痛点②同一时刻一起巡检会差一拍:第 2 个还在改、第 3 个同时扫到的是"没改",白扫。职责不同、还可能有先后依赖,该错开 → 新增 **`CodeAgent.loopMins`(每 Agent 自己的巡检间隔,留空=用全局)**。Agent 卡片上加一格"巡检每 [N] 分钟"(实现位显示"空闲即续、不按间隔",不渲染输入)。runCoding 里非实现位用 `a.loopMins ?? code.loopMins`,可把下游设得比上游晚,让它在上游改完后才扫。

### 协作编程：重启后续上次进度、不重复工作
- 痛点：重启后 PTY 全没了，重新「运行」会让 Agent **从头再做一遍**（重复工作）。
- 修（靠 prompt + TEAM_NOTES.md 做进度，不另存状态）：
  - 主写初始指令开头加"先过一遍现有代码 + TEAM_NOTES.md 判断做到哪了，**若已做一部分就从断点续、绝不重做**，全新才从头"；且每完成一步**先把这步记进 TEAM_NOTES.md 当进度清单**，重启后据此续。
  - 审查/测试初始指令加"可能是续上次、TEAM_NOTES.md 里提过的别重复提、只补新问题"。
- 任务配置本就存在 `council.code`（重开自动填回表单），所以重启后直接再点「运行」即续。

### 协作编程：主写不肯继续下一步 → 修
- 现象：主写干完一步就停（"Churned for Xs"）、交还控制，得手动打字催「继续下一步」。根因两条：
  1. **持续协作没开**（loop=false）就没有任何定时器去催它——这类 CLI 干完一回合就停，天性如此，必须有东西再戳。开关打开才有催。
  2. 初始指令写的是"先把**基础实现**做出来"——本身在暗示做完雏形就停。
- 修：① 初始 isImpl 指令改成"一步步把**整个任务**做完，不要只做基础雏形就停，做完一步紧接着下一步直到真正完成"。② 持续协作里**主写空闲就立刻续**（minGap 固定 15s，不等 loopMins）；loopMins 那个间隔只管审查/测试的周期巡检。

### 协作编程·历史任务（多份配置可回填）
- 痛点：协作编程只存「当前这一份」配置（`council.code`），换个新任务就把旧的覆盖了，**之前的任务在历史里找不到**（而「运行历史」只记流水线/圆桌/GEO 这种出一份结果的模式，协作编程/终端是实时终端、不进历史）。
- 新增 `council.codehist`：每次「运行」前 `pushCodeTask()` 把当前配置（目录 + 任务 + 协作约定 + agents + auto/loop/loopMins）快照存一份（最近 30 份，同 dir+task 去重并置顶）。
- 任务标签旁加「📋 历史任务」按钮 → `#code-hist-modal` 列表：每条显示标题（任务首行）+ 时间 + 目录 + 参与 CLI；**点一条即把那份配置填回表单**（深拷贝 agents），可单条 ✕ 删或「清空」。

### 协作编程「持续协作」无限循环（空闲触发 + 短指令）
- 新增 `code.loop` 开关（表单复选框「持续协作」，默认关）+ `code.loopMins`（间隔分钟，输入框 `#code-loop-mins`，默认 2.5，0.5~60 可调）：开启后协作编程不再跑完一轮就停。
- **只在 Agent 空闲时巡检**（之前是固定 `setInterval` 盲发，会插进 Agent 正忙/用户正打字时）：Term 上记 `lastOutputAt`（PTY 有输出就更新，TUI 忙时一直在重绘动画）和 `lastInputAt`（用户键入就更新）。计时器改成每 10s 轮询，只有「距上次巡检 ≥ loopMins」且「无输出 ≥8s（已干完这轮）」且「用户 ≥12s 没打字」三条都满足才发。
- **短指令、不再打一大段**：Agent 启动时已知道自己的职责 + TEAM_NOTES.md 约定，巡检只发一句短的。主写（isImpl）发"先看 TEAM_NOTES.md，有反馈就先改；**审查/测试没提修改意见就直接继续做任务的下一步、别等**；全部做完才回'全部完成'"——后边两个没意见时主写不卡住、继续推进；审查/测试发"再巡检一遍，有问题处理并写进 TEAM_NOTES.md，没有回'暂无'"。
- **修卡住**：之前 `write_pty(text + "\r")` 一次发，TUI 的粘贴处理会把结尾 `\r` 吞掉，整行卡在输入框里不提交。改成先发文本、350ms 后**单独再发一个 `\r`**才真正回车提交。
- 计时器存 `codeLoopTimers`；停止：取消勾选（清计时器）、关终端（term.sessionId 空时自清）、重跑（resetPanes 里 clearCodeLoop）。⚠️ = 持续耗 token。

### 圆桌/流水线：codex / gemini 调不动的根因 + 修复
- 这俩 CLI 在 **headless（圆桌/流水线走 `cli_run` 管道、无 tty、在 app 工作目录）+ 非信任目录** 下会拒绝运行：codex 报 `Not inside a trusted directory … --skip-git-repo-check`，gemini 报 `not running in a trusted directory … --skip-trust`。

### 协作编程：claude 启动卡在 bypass 确认 → 自动接受
- `claude --dangerously-skip-permissions` 交互启动时弹一次性 "Bypass Permissions … ❯1. No / 2. Yes, I accept" 菜单，默认在「No」，不选就一直等（看着像启动不了）。
- 修复：**监听 PTY 输出**，检测到菜单标志词（`confirm` + `exit`，因为 "Yes, I accept" 被光标转义码拆开匹配不到）后，自动发「↓」再「回车」选中「2. Yes」。靠"菜单已渲染"驱动、不是盲发定时（盲发会落在默认 No 上把它选退出）。只在「自动批准」勾选时生效。

### 流水线：依赖感知并行执行（借 gstack autoplan 的洞见）
- 之前 `run()` 用 `for` 串行跑所有步骤，即使第 2/3/4 步只引用 `{{input}}`（各自独立分析同一输入）也被一个个跑。
- 新增纯函数 `stepDeps(tpl, idx)`（utils.ts，含测试）：从模板算上游依赖——`{{prev}}`→idx-1、`{{N}}`→N-1（只算 < idx 的反向引用），只有 `{{input}}` = 无依赖。
- runSteps 改成**按依赖分波并行**：先建好全部卡片（保持顺序，排队中），每波跑所有"依赖已完成"的步骤（Promise.all 并发，各自 stream）。纯链式（每步 `{{prev}}`）自动退化全串行，零回归；扇出型省时间。
- 顺带更健壮：某步出错只跳过依赖它的下游（transitive），独立分支照常跑完——旧代码是一出错 `break` 整条停。
